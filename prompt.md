# Novel Generator — Prompt 设计文档

> 版本：v1.0
> 适用题材：言情/古言/总裁、悬疑/推理/惊悚、玄幻/仙侠/奇幻、都市/现代
> 目标体量：1-3 万字短篇，10-15 章，约 2000 字/章
> 流水线：设定 → 大纲 → 正文 → 摘要压缩

## 0. 全局角色设定（System Prompt）

所有阶段调用 LLM 时，统一注入此 system prompt 作为基础人格。

```
你是一名专业网络小说作家，擅长在七猫、番茄等平台创作 1-3 万字短篇。
你深谙网文节奏：3 章一小爽、5 章一中爽、10 章一大爽，开篇 500 字内必须抛出钩子。
你的语言风格简洁有画面感，对话推动情节，避免大段心理描写和环境铺陈。
你严格按照用户给出的设定、大纲、前情摘要进行创作，不擅自改写主线。
你的输出必须严格遵循指定的格式约束（JSON / 纯文本 / Markdown），不添加任何解释性文字。
```

**调用参数建议：**

| 阶段 | temperature | top_p | max_tokens |
|---|---|---|---|
| 设定生成 | 0.9 | 0.95 | 2000 |
| 章纲生成 | 0.8 | 0.9 | 4000 |
| 正文生成 | 0.85 | 0.92 | 4000 |
| 摘要压缩 | 0.3 | 0.8 | 800 |

## 1. 阶段一：故事设定生成

### 1.1 用途

根据用户输入的「题材 + 一句话灵感」，扩展出一份完整的故事设定档案。这份档案是后续所有阶段的根基，会作为 context 注入大纲和正文生成。

### 1.2 输入

```json
{
  "genre": "都市/现代",
  "subgenre": "总裁追妻",
  "inspiration": "一夜情之后发现对方是自己新上任的顶头上司",
  "target_chapters": 12,
  "target_words_per_chapter": 2000
}
```

### 1.3 Prompt 模板

```
【任务】
基于以下题材与灵感，生成一份完整的短篇小说设定档案。

【题材】{{genre}} - {{subgenre}}
【灵感】{{inspiration}}
【篇幅】共 {{target_chapters}} 章，约 {{target_words_per_chapter}} 字/章

【要求】
1. 主角设定要有"反差感"和"代入感"，不要完美无缺
2. 至少包含 1 个核心冲突 + 2 个次要冲突
3. 世界观规模匹配短篇体量，不要构建过于宏大的设定
4. 主角的核心目标必须在 12 章内可达成

【输出格式】严格按以下 JSON 结构输出，不要添加任何解释：

{
  "title": "小说标题（6-12 字，有钩子）",
  "logline": "一句话故事梗概（30 字内）",
  "tone": "整体基调（如：甜虐交织、热血爽文、暗黑悬疑）",
  "protagonist": {
    "name": "姓名",
    "age": 0,
    "identity": "身份职业",
    "personality": "性格关键词 3-5 个",
    "motivation": "核心动机",
    "flaw": "主要缺陷或弱点"
  },
  "antagonist": {
    "name": "姓名",
    "identity": "身份",
    "goal": "对立目标",
    "relationship": "与主角的关系"
  },
  "supporting_cast": [
    {"name": "", "role": "（如：闺蜜/师父/反派打手）", "function": "在故事中的功能"}
  ],
  "world_setting": {
    "time_space": "时空背景一句话",
    "key_rules": ["世界观核心规则 1", "规则 2"],
    "key_locations": ["主要场景 1", "场景 2", "场景 3"]
  },
  "core_conflict": "核心冲突（贯穿全篇）",
  "secondary_conflicts": ["次要冲突 1", "次要冲突 2"],
  "story_arc": {
    "opening_hook": "开篇 500 字内的钩子事件",
    "midpoint_twist": "中段（第 6-7 章）的关键反转",
    "climax": "高潮（倒数第 2 章）的决定性场面",
    "resolution": "结局收束方式"
  },
  "selling_points": ["卖点 1（如：先婚后爱）", "卖点 2", "卖点 3"]
}
```

### 1.4 后处理

- 校验 JSON 格式合法性，失败则重试（最多 2 次）
- 校验必填字段完整性
- 用户确认 / 修改后入库，作为 `Project.setting` 字段

## 2. 阶段二：章纲生成

### 2.1 用途

基于已确认的设定档案，生成全本 N 章的章纲。每章包含场景、目标、冲突、爽点、钩子，是正文生成的直接依据。

### 2.2 节奏模板（写入 prompt 作为约束）

```
【12 章短篇标准节奏】
第 1 章：开篇钩子 + 主角登场 + 抛出核心冲突
第 2 章：建立日常 + 引入关键配角 + 第一个小冲突
第 3 章：第一个小爽点 + 揭示部分世界观
第 4 章：冲突升级 + 主角陷入困境
第 5 章：中场小爽点 + 反派初次正面交锋
第 6 章：中段反转 + 关系/格局发生质变
第 7 章：主角最低谷 + 误会/背叛/失败
第 8 章：转机出现 + 主角觉醒/获得关键资源
第 9 章：反击开始 + 中爽点
第 10 章：决战前夜 + 所有伏笔回收准备
第 11 章：高潮决战 + 大爽点
第 12 章：结局收束 + 情感升华 / 留白
```

### 2.3 Prompt 模板

```
【任务】
基于以下故事设定，为这部 {{target_chapters}} 章短篇生成完整章纲。

【故事设定】
{{setting_json}}

【节奏要求】
{{pacing_template}}

【强制要求】
1. 必须输出 {{target_chapters}} 章，不多不少
2. 每章末尾必须留钩子（悬念/反转/情绪爆点），引导读者读下一章
3. 主线冲突必须贯穿，每章至少推进一次
4. 第 1 章前 500 字必须有强钩子，禁止平铺直叙
5. 中段反转（第 6-7 章）必须改变主角处境
6. 高潮章节（倒数第 2 章）必须有"决定性场面"
7. 不要在章纲里写出完整对话，只写情节走向

【输出格式】严格按以下 JSON 数组输出，每个元素为一章：

[
  {
    "index": 1,
    "title": "章节标题（8-15 字，有画面感或悬念）",
    "scene": "主场景（一句话，时间+地点）",
    "characters": ["出场角色 1", "角色 2"],
    "goal": "本章主角想要达成的目标",
    "conflict": "本章冲突点（与谁/与什么对抗）",
    "key_events": [
      "关键事件 1（按时间顺序）",
      "关键事件 2",
      "关键事件 3"
    ],
    "emotional_beat": "本章情绪基调（紧张/甜蜜/愤怒/绝望等）",
    "payoff": "本章爽点 / 信息揭示 / 情感推进（至少 1 个）",
    "ending_hook": "章末钩子（具体描述，必须能引导读者读下一章）",
    "foreshadowing": ["埋下的伏笔（可空数组）"],
    "callback": ["回收的伏笔（前几章埋的，可空数组）"]
  }
]
```

### 2.4 后处理

- 校验：章数是否等于 `target_chapters`、index 连续、每章有 ending_hook
- 在 Web 界面以表格形式展示，用户可：
  - 单章重新生成（保留前后章不变）
  - 手工编辑任意字段
  - 调整章节顺序 / 增删章节
- 确认后入库，作为 `Outline` 表，每章一条记录

### 2.5 单章重新生成的子 Prompt

```
【任务】
重新生成第 {{index}} 章的章纲。

【故事设定】{{setting_json}}
【全本章纲】{{full_outline_json}}
【上一章纲要】{{prev_chapter_json}}
【下一章纲要】{{next_chapter_json}}
【用户的修改诉求】{{user_feedback}}

【要求】
- 必须与前后章保持因果连续
- 必须满足用户修改诉求
- 输出格式同章纲单元（单个 JSON 对象）
```

## 3. 阶段三：章节正文生成

### 3.1 用途

基于设定、当前章纲、前文摘要，生成本章完整正文（约 2000 字）。这是流水线最关键也最容易"跑偏"的环节，prompt 设计需要严格约束。

### 3.2 上下文组装策略

为避免 token 爆炸 + 保证一致性，每次只注入：

| 内容 | 来源 | 用途 |
|---|---|---|
| 故事设定（精简版） | `Project.setting` 关键字段 | 人物、世界观锚点 |
| 全本章纲（标题+一句话摘要） | `Outline` 表 | 全局位置感 |
| 当前章纲（完整） | `Outline[index]` | 本章蓝图 |
| 上一章正文末 500 字 | `Chapter[index-1].content` 尾部 | 衔接自然 |
| 前文累积摘要 | `Memory.cumulative_summary` | 长程一致性 |
| 已埋伏笔列表 | `Memory.foreshadowing` | 伏笔追踪 |

### 3.3 Prompt 模板

```
【任务】
基于以下信息创作第 {{index}} 章正文，目标字数 {{target_words}} 字（±10% 可接受）。

【故事设定】
{{setting_compact}}

【全本章纲速览】
{{outline_summary}}

【本章详细蓝图】
{{current_outline_json}}

【上一章末尾衔接】
{{prev_chapter_tail}}

【前文累积摘要】
{{cumulative_summary}}

【待回收伏笔】
{{pending_foreshadowing}}

【创作要求】
1. 严格按本章蓝图的 key_events 顺序推进，不擅自增删主线事件
2. 必须实现蓝图中的 payoff（爽点/信息揭示/情感推进）
3. 章末必须落在蓝图指定的 ending_hook 上
4. 用 60% 篇幅写场景动作和对话，30% 心理与画面，10% 必要交代
5. 对话要符合人物身份和性格设定，避免 AI 腔（"我感受到了一种复杂的情绪"等）
6. 段落短促，单段不超过 4 行，单句不超过 30 字
7. 字数控制在 {{target_words}} 字左右
8. 不写"上一章…"、"接下来…"等元叙述
9. 不输出标题（标题已由系统管理）

【禁止事项】
- 禁止使用"突然"、"忽然"、"不经意间"作为转折开头超过 1 次
- 禁止 AI 常见套话："命运的齿轮开始转动"、"故事才刚刚开始"等
- 禁止角色性格突变（必须与设定一致）
- 禁止破坏世界观规则
- 禁止剧透后续章节内容

【输出格式】纯文本正文，直接开始，不加任何前后缀、不加标题、不加 Markdown 标记。
```

### 3.4 重生成 / 续写场景

**整章重写**（用户对当前章不满意）：

```
【任务】
重写第 {{index}} 章。

【上一版正文】{{previous_version}}
【用户反馈】{{user_feedback}}
【保留要素】{{keep_elements}}

[其余 context 同上]

【要求】
- 严格响应用户反馈进行调整
- 保留 {{keep_elements}} 中列出的情节和场景
- 不得违反章纲蓝图的核心走向
```

**局部续写**（用户在某段后插入「继续」）：

```
【任务】
从以下断点继续创作本章后续内容。

【已有正文】{{content_so_far}}
【断点位置】{{cursor_position}}
【后续方向提示】{{user_hint}}

[其余 context 同上]

【要求】
- 续写部分必须与前文风格、视角、时态一致
- 续写完成后本章总字数应达到 {{target_words}}
```

## 4. 阶段四：章节摘要压缩

### 4.1 用途

每章正文生成完后，立即调用此 prompt 生成本章结构化摘要，并累积到 `Memory.cumulative_summary`。这是支撑「长程一致性」的核心：后续章节生成时，不需要再喂全文，只需喂摘要 + 末尾 500 字。

### 4.2 Prompt 模板

```
【任务】
将以下章节正文压缩成结构化摘要。

【正文】
{{chapter_content}}

【本章蓝图】
{{current_outline_json}}

【输出格式】严格按以下 JSON 结构，不要添加解释：

{
  "index": {{index}},
  "one_line_summary": "一句话概括本章（30 字内）",
  "key_facts": [
    "本章发生的关键事实 1（影响后续剧情的）",
    "事实 2",
    "事实 3"
  ],
  "character_state_changes": [
    {"name": "角色名", "change": "状态/关系/认知发生的变化"}
  ],
  "new_foreshadowing": ["本章新埋的伏笔"],
  "resolved_foreshadowing": ["本章回收的伏笔"],
  "world_state_delta": "世界观/格局发生的变化（无则填 null）",
  "emotional_residue": "本章留给读者的情绪余韵",
  "next_chapter_setup": "为下一章铺垫的关键状态"
}
```

### 4.3 累积摘要的拼接策略

`cumulative_summary` 不是简单拼接所有章节的 `one_line_summary`，而是采用**滚动压缩**：

- 每 3 章触发一次"中期合并"：把第 N-2、N-1、N 三章的摘要合并成一段「阶段叙事」
- 每 6 章触发一次"全局压缩"：保留所有 `character_state_changes` 累计、`pending_foreshadowing` 全量、关键剧情节点列表
- 写入正文 prompt 时，按需截取最近 N 章的中期合并 + 全局压缩

**中期合并 Prompt：**

```
【任务】将以下 3 章摘要合并为一段连贯的"阶段叙事"，控制在 200 字内。
【章节摘要】{{three_summaries}}
【输出】纯文本，一段话，不加标题。
```

## 5. 题材风格补充

每个题材在阶段三正文 prompt 中追加一段「风格附加约束」。

### 5.1 言情/古言/总裁

```
【风格约束】
- 视角：女主第一/第三人称限制视角
- 情感线密度：每章至少 2 次男女主交锋（眼神/对话/肢体接触）
- 男主刻画：通过细节而非心理描写（皱眉、停顿、冷笑、不动声色的关心）
- 古言：避免现代词汇（OK、emo、靠谱），称谓符合身份
- 总裁：商战桥段不要过度专业，重在权力压制感和情感张力
```

### 5.2 悬疑/推理/惊悚

```
【风格约束】
- 视角：第三人称受限或第一人称回忆体
- 信息节奏：每章至少抛 1 个新疑点 + 解 1 个旧疑点（净增量为正）
- 公平原则：所有线索必须在揭示前有过出现（哪怕一笔带过）
- 氛围词汇：偏冷色调、感官描写（声音、气味、温度）
- 禁止：上帝视角剧透、关键线索靠"突然想起"
```

### 5.3 玄幻/仙侠/奇幻

```
【风格约束】
- 视角：男主第三人称跟随
- 等级体系：每次出现新境界/法宝必须给出对比锚点（与已知等级对照）
- 战斗描写：招式名 + 物理感受 + 场景破坏，避免纯文字招式罗列
- 升级节奏：不能章章升级，要"压抑→爆发"循环
- 古风词汇：使用"罢了""倒是""怎可"等，但避免文言文堆砌
```

### 5.4 都市/现代

```
【风格约束】
- 视角：主角第一/第三人称
- 真实感：行业细节要经得起推敲（不必专业但不能离谱）
- 对话：贴近当代年轻人语感，可适度网络用语但不滥用
- 节奏：开篇直入，避免大段背景铺垫
- 爽点：实力打脸 / 反转身份 / 资源碾压，但不要过度玛丽苏
```

## 6. 通用约束与反模式

### 6.1 全局禁用词与套路

无论哪个阶段，以下内容都应在 prompt 中强制禁用：

```
【全局禁用】
1. AI 套话：
   - "命运的齿轮"、"故事才刚刚开始"、"这一切的背后"
   - "他/她不知道的是"（上帝视角剧透）
   - "在这个世界上"、"在那个瞬间"
2. 空洞修辞：
   - "无法言喻的"、"难以形容的"、"复杂的情绪"
   - "时间仿佛静止"、"心如刀绞"、"五味杂陈"
3. 描写偷懒：
   - "美得像画一样"、"帅得无法形容"
   - "气氛瞬间凝固"、"空气仿佛冻结"
4. 转场敷衍：
   - "时间一晃而过"、"几天后"（除非明确需要跳跃）
   - 同一章内不超过 2 次"突然/忽然"
```

### 6.2 输出格式校验规则

| 阶段 | 期望格式 | 校验项 | 失败处理 |
|---|---|---|---|
| 设定 | JSON | 字段完整、类型正确 | 自动重试 2 次，仍失败则报错给用户 |
| 章纲 | JSON 数组 | 长度 = target_chapters，index 连续 | 自动重试 2 次 |
| 正文 | 纯文本 | 字数 ±15% 容忍，无 JSON/Markdown 残留 | 字数偏差大触发"补写/裁剪"子流程 |
| 摘要 | JSON | 字段完整 | 自动重试 1 次 |

### 6.3 字数偏差修正子流程

**字数过少（< target × 0.85）：**

```
【任务】对以下正文进行扩写，使总字数达到 {{target_words}} 字。
【已有正文】{{content}}
【扩写要求】
- 不增加新情节，只丰富现有场景的细节、动作、对话、心理
- 保持原有节奏和钩子
- 重点扩充：场景描写、对话张力、关键动作的慢镜头
```

**字数过多（> target × 1.15）：**

```
【任务】对以下正文进行精简，使总字数控制在 {{target_words}} 字内。
【已有正文】{{content}}
【精简要求】
- 优先删除：冗余形容词、重复心理活动、与主线无关的场景
- 保留：关键事件、核心对话、ending_hook
- 不删除任何 key_events 中列出的情节节点
```

### 6.4 重试与降级策略

```
LLM 调用失败 / 校验失败的处理顺序：
1. 同 prompt 重试（最多 2 次，指数退避 1s/3s）
2. 降低 temperature 重试一次（temperature - 0.2）
3. 仍失败 → 在 SSE 推送错误事件 → 用户介入
```

### 6.5 一致性兜底检查（可选，2.0）

每章正文生成后，可追加一次"一致性校验调用"：

```
【任务】检查以下正文是否违反故事设定或前文事实，输出问题列表。
【设定】{{setting}}
【前文累积事实】{{key_facts_all}}
【本章正文】{{content}}
【输出格式】
{
  "consistency_ok": true/false,
  "issues": [
    {"type": "人物性格冲突/世界观违反/事实矛盾", "detail": "具体问题"}
  ]
}
```

发现问题则触发整章重写 + 在 prompt 中补充「需规避以下问题」。
